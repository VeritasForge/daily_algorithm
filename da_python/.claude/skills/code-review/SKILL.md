# 알고리즘 코드 리뷰 + 피드백 반영

대상 파일: ${1:-$CURRENT_FILE}
리뷰 포커스: ${2:-전체}

---

## Phase 1: 리뷰

### Step 1. 파일 읽기

1. **대상 파일** 읽기: 위에서 결정된 파일
2. **테스트 파일** 탐색 및 읽기:
   - `src/leetcode/{name}.py` → `test/leetcode/test_{name}.py`
   - `src/etc/{name}.py` → `test/etc/test_{name}.py`
   - `src/common/{name}.py` → `test/common/test_{name}.py`
3. **git diff** 확인: `git diff {대상 파일}` — 수정 내역이 있으면 변경점 파악

### Step 2. 5가지 항목 리뷰

CLAUDE.md의 답변 원칙(맥락 고려, 다양한 관점, 성장 고려, 단계별 설명, 예시 사용, 대안 제시, 현실 비유, 논리 공개)을 적용하여 리뷰한다.

리뷰포커스(${2})가 지정된 경우 해당 항목에 특히 중점을 둔다.
각 항목에 대해 구체적인 개선 방안을 코드 예시와 함께 제시한다.

#### 1. 알고리즘 정확성
- 문제 요구사항을 완전히 만족하는가?
- 모든 엣지 케이스를 처리하는가?

#### 2. 복잡도 분석
- **시간복잡도**: Big-O 분석
- **공간복잡도**: Big-O 분석
- 최악/평균/최선의 경우 분석 (해당되는 경우)

#### 3. 코드 품질
- 타입 힌트: PEP 484 준수 (Python 3.9+ 스타일)
  - `list[int]` (O) vs `List[int]` (X)
  - `TreeNode | None` (O) vs `Optional[TreeNode]` (X)
- mypy 타입 체크 통과 여부
- 변수명과 함수명의 명확성

#### 4. Pythonic 개선사항
- 내장 함수 활용 (map, filter, list comprehension 등)
- 더 간결하고 가독성 좋은 표현 방법

#### 5. 다양한 풀이법
현재 솔루션 외에 다른 접근 방법이 있으면 제시:
- 브루트포스 vs 최적화된 솔루션
- 시간-공간 트레이드오프

### Step 3. code-reviewer 서브에이전트 (조건부)

아래 **호출 조건** 중 하나 이상 해당하면 `superpowers:code-reviewer` 서브에이전트를 배치한다.

**호출 조건:**
- 탐욕법, DP, 분할정복 등 정당성 증명이 필요한 알고리즘
- 기존 풀이를 리팩토링/단순화한 경우 (동치성 검증)
- 사용자가 명시적으로 요청한 경우

**스킵 조건:**
- 단순 구현 (brute force, 직관적 풀이)
- 표준 패턴 적용 (two pointer, sliding window 등 정형화된 풀이)

서브에이전트 배치 시 전달할 컨텍스트:
- 대상 파일의 전체 코드
- git diff (있는 경우)
- 테스트 파일 내용
- `uv run pytest {테스트파일} -v` 실행 결과

## Phase 2: 반영

Phase 1의 리뷰 결과(자체 리뷰 + code-reviewer 결과)를 바탕으로 피드백을 반영한다.

### Step 4. 피드백 분류

리뷰 결과를 다음 두 가지로 분류한다:

| 분류 | 기준 | 반영 위치 | 예시 |
|------|------|----------|------|
| 🔧 코드 개선 | 정확성, 성능, 코드 품질 관련 | 코드 직접 수정 | 타입 힌트 추가, 변수명 개선, dead code 제거 |
| 📝 학습 포인트 | 알고리즘 이해, 패턴, 사고방식 관련 | 독스트링 업데이트 | 핵심 아이디어 수정, 적용 패턴 추가, 교환 논증 보강 |

분류할 피드백이 없으면 "✅ 개선사항 없음 — 현재 코드와 독스트링이 적절합니다."를 출력하고 종료한다.

### Step 5. 수정안 제시 및 사용자 승인

분류된 피드백을 기반으로 구체적인 수정안을 생성하여 사용자에게 보여준다.

출력 형식:

```
🔧 코드 개선 (N건):
  1. {위치}: {변경 내용 요약}
  2. {위치}: {변경 내용 요약}

📝 학습 포인트 (M건):
  1. {영역}: {변경 내용 요약}
  2. {영역}: {변경 내용 요약}
```

각 수정안에 대해 변경 전/후를 보여준다.

AskUserQuestion으로 사용자에게 승인을 요청한다:
- **전체 승인**: 모든 수정안 적용
- **항목별 선택**: 원하는 것만 적용
- **스킵**: Phase 2를 건너뜀

### Step 6. 승인된 항목 적용

사용자가 승인한 수정사항을 코드에 적용한다.

### Step 7. 테스트 검증

수정 후 테스트를 실행하여 기존 테스트가 깨지지 않았는지 확인한다.

Run: `uv run pytest {테스트파일} -v`

테스트 실패 시:
1. 실패 원인을 분석하여 사용자에게 보고한다
2. 수정사항을 롤백할지 사용자에게 묻는다

## 독스트링 업데이트 규칙

📝 학습 포인트를 독스트링에 반영할 때 다음 규칙을 따른다:

### 수정 가능 영역

```python
"""
문제 설명: ...        # ← 절대 수정하지 않음 (문제 원문)

핵심 아이디어:        # ← 설명이 현재 구현과 불일치하면 수정
    ...
    탐욕법 교환 논증:  # ← 최적성 근거가 부족하면 보강
        ...
    적용 패턴:        # ← 재사용 가능한 패턴이 발견되면 추가
        ...
"""

def solution(...):
    """함수 독스트링  # ← 누락/불완전하면 추가/수정
    Args/Returns/Examples
    """
```

### 규칙

1. **문제 설명(원문)은 절대 수정하지 않는다**
2. 핵심 아이디어가 현재 구현과 일치하면 건드리지 않는다
3. 적용 패턴은 리뷰에서 일반화할 만한 인사이트가 있을 때만 추가한다
4. 함수 독스트링이 이미 완전하면 건드리지 않는다
5. 독스트링 추가 시 기존 프로젝트 컨벤션(CLAUDE.md)을 따른다
